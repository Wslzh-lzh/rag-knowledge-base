## 1. 系统架构与设计模式

### 1.1 Provider 模式（策略模式）

**知识点讲解：**

Provider 模式（也叫策略模式）是一种行为设计模式，它定义了一系列算法，把每个算法封装起来，并且使它们可以互相替换。这个模式让算法的变化独立于使用算法的客户。

在实际开发中，Provider 模式的核心是：
- 定义一个抽象基类（Base Class），规定统一的接口方法
- 不同的实现类继承这个基类，各自实现具体逻辑
- 有一个 Router/Factory 类，根据配置选择使用哪个 Provider
- 调用方只依赖抽象基类，不依赖具体实现

**优点：**
- 可扩展性好，新增一种实现只需加一个 Provider 类
- 可以在运行时切换不同的实现
- 符合开闭原则（对扩展开放，对修改关闭）
- 便于单元测试，可以用 Mock 实现替换真实服务

**在本项目中的应用：**

项目中 5 个核心组件都采用了 Provider 模式：

1. **Embedding Provider** — 嵌入模型
   - 抽象基类：`BaseEmbeddingProvider`，定义 `embed()` 和 `embed_batch()` 方法
   - 实现：`DashScopeEmbeddingProvider`（真实调用）、`MockEmbeddingProvider`（本地模拟）
   - 路由：`embeddings/router.py` 根据 `DEFAULT_EMBEDDING_PROVIDER` 配置选择

2. **Vector Store Provider** — 向量存储
   - 抽象基类：`BaseVectorStore`，定义 `add()`、`search()`、`delete()` 等方法
   - 实现：`QdrantVectorStore`（生产用）、`MemoryVectorStore`（内存存储，测试用）
   - 路由：`vector_store/router.py` 根据 `VECTOR_STORE_BACKEND` 配置选择

3. **Full-Text Search Provider** — 全文检索
   - 抽象基类：`BaseFulltextSearcher`，定义 `index()`、`search()`、`delete()` 等方法
   - 实现：`PostgreSQLSearcher`、`OpenSearchSearcher`
   - 路由：`fulltext/router.py` 根据 `FULLTEXT_SEARCH_BACKEND` 配置选择

4. **LLM Provider** — 大语言模型
   - 抽象基类：`BaseLLMProvider`，定义 `chat()`、`chat_stream()` 等方法
   - 实现：`OpenAIClient`（OpenAI 兼容接口）、`EchoProvider`（回显模式，测试用）
   - 路由：`llm/router.py` 根据 `DEFAULT_LLM_PROVIDER` 配置选择

5. **Reranker Provider** — 重排器
   - 抽象基类：`BaseReranker`，定义 `rerank()` 方法
   - 实现：`DashScopeReranker`、`MockReranker`
   - 路由：`rerankers/router.py` 根据 `DEFAULT_RERANKER_PROVIDER` 配置选择

**实际应用场景：**
- 本地开发时，用 Mock/Memory/Echo Provider，不需要真实 API Key，快速开发
- 生产环境时，切换到真实服务（DashScope/Qdrant/OpenSearch）
- 接入新的模型服务时，只需要新增一个 Provider 类，不用改业务代码

### 1.2 前后端分离架构

**知识点讲解：**

前后端分离是一种软件架构模式，前端和后端作为独立的两个项目开发和部署，通过 API 进行数据交互。

核心特点：
- 前端负责页面展示和用户交互
- 后端负责业务逻辑和数据处理
- 通过 RESTful API 或 GraphQL 进行通信
- 前后端可以独立开发、独立部署、独立扩展

**优点：**
- 前后端团队各司其职，并行开发
- 前端可以适配多端（Web、移动端、小程序）
- 技术选型灵活，前后端可以用不同的语言和框架
- 便于维护和迭代

**在本项目中的应用：**

- **后端**：FastAPI 提供 RESTful API，返回 JSON 数据，路径前缀 `/api/v1`
- **前端**：Next.js 14 构建页面，通过 `fetch` 调用后端 API
- **通信协议**：HTTP + JSON，流式问答用 SSE（Server-Sent Events）
- **认证方式**：JWT Token，前端存在 localStorage，每次请求带在 Header 里

## 2. RAG 技术与检索优化

### 2.1 RAG 基础概念

**知识点讲解：**

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大语言模型结合的技术。它的核心思想是：在生成回答之前，先从外部知识库中检索相关的文档片段，然后将这些片段作为上下文一起喂给 LLM，让 LLM 基于检索到的信息生成回答。

**为什么需要 RAG：**
- LLM 的知识是训练时的，有截止日期，不知道最新信息
- LLM 容易产生幻觉（编造事实），RAG 可以让回答基于真实文档
- 企业内部的私有数据不可能都喂给 LLM 训练，RAG 可以直接接私有知识库
- 相比微调成本更低，更新知识不需要重新训练模型

**RAG 的基本流程：**
1. 文档上传 → 解析 → 分块 → 向量化 → 存入向量库（离线阶段）
2. 用户提问 → 问题向量化 → 从向量库检索相关片段 → 构建 Prompt → LLM 生成回答（在线阶段）

**在本项目中的应用：**

离线阶段（文档摄取）：
```
用户上传文档 → 解析文本 → 文本清洗 → 语义分块 → Embedding 向量化 → 存入 Qdrant + 全文索引
```

在线阶段（问答流程）：
```
用户提问 → 问题向量化 → 混合检索（向量+全文）→ RRF 融合 → Rerank 重排 → 构建上下文 Prompt → LLM 生成 → SSE 流式输出
```

---

### 2.2 向量检索

**知识点讲解：**

向量检索（Vector Search）是一种基于向量相似度的检索方式。它的原理是把文本转换成向量（Embedding），然后计算查询向量和文档向量之间的相似度（通常用余弦相似度），找出最相似的 Top K 个文档。

**关键概念：**
- **Embedding（嵌入）**：把文本转换成固定维度的向量，语义相近的文本向量距离近
- **余弦相似度**：衡量两个向量方向的相似程度，值越接近 1 越相似
- **向量数据库**：专门存储和检索向量的数据库，支持近似最近邻（ANN）搜索，速度比暴力计算快很多

**常用的向量数据库：**
- Qdrant — Rust 写的，性能好，支持本地模式
- Milvus — 功能全面，分布式架构
- Pinecone — 托管服务，开箱即用
- Chroma — 轻量级，适合原型开发

**在本项目中的应用：**

- 用 Qdrant 作为向量数据库，支持本地模式（存文件）和服务端模式
- Embedding 用 DashScope 的 text-embedding-v3 模型，维度 1024
- 每个文档分块后生成向量，存在 Qdrant 里，payload 里存 chunk_id、kb_id 等元数据
- 检索时用 cosine 相似度，返回 Top K 个最相关的 chunk

---

### 2.3 全文检索

**知识点讲解：**

全文检索（Full-Text Search）是一种基于关键词匹配的检索方式，它会对文本进行分词，建立倒排索引，查询时根据关键词匹配度排序返回结果。

**关键概念：**
- **分词（Tokenization）**：把文本切成一个个词（token）
  - 英文：按空格、标点切分，比较简单
  - 中文：需要专门的分词器，比如 IK 分词器、结巴分词
- **倒排索引（Inverted Index）**：记录每个词出现在哪些文档里的数据结构，类似书后面的索引
- **相关性评分**：根据词频、文档频率等计算文档和查询的相关度，常用算法有 BM25

**为什么向量检索还要配全文检索？**
- 向量检索擅长语义匹配，但可能漏掉精确的关键词匹配
- 全文检索擅长精确匹配，但对同义词、语义相近的词不行
- 两者互补，混合使用效果更好

**在本项目中的应用：**

支持两种全文检索后端：
1. **PostgreSQL 全文检索** — 内置的 tsvector + tsquery，简单够用，不需要额外服务
2. **OpenSearch** — 专业的搜索引擎，支持 IK 分词器，性能更好，功能更全

中文分词处理：
- OpenSearch 检测到 IK 分词插件时索引用 `ik_max_word`（细粒度切分），搜索用 `ik_smart`（智能切分）
- 没有 IK 插件时降级用 standard 标准分词器
- PostgreSQL 用中文分词扩展或简单的 n-gram 方式

---

### 2.4 混合检索与 RRF 融合

**知识点讲解：**

混合检索（Hybrid Search）就是同时用向量检索和全文检索，然后把两路结果合并起来。这样可以同时利用语义匹配和关键词匹配的优势。

**RRF（Reciprocal Rank Fusion，倒数排名融合）** 是一种常用的结果融合算法。它的思想很简单：
- 每个检索结果根据排名给一个分数：`1 / (k + rank)`，k 是一个常数（通常取 60）
- 同一个文档在两路结果中都出现的话，分数相加
- 最后按总分排序

**RRF 公式：**
```
score(d) = Σ 1 / (k + rank_i(d))
```
- rank_i(d) 是文档 d 在第 i 个检索结果列表中的排名
- k 是常数，通常取 60

**为什么用 RRF 而不是直接加权平均相似度？**
- 向量检索的相似度分数和全文检索的 BM25 分数量级不一样，不能直接加权
- RRF 只看排名，不看具体分数，避免了不同检索方式分数不可比的问题
- 实现简单，效果稳定

**在本项目中的应用：**

在 `services/retrieval/hybrid.py` 中实现了混合检索：
1. 同时调用向量检索和全文检索，各返回 Top N 个结果
2. 用 RRF 算法融合两路结果的排名
3. 融合后再送去 Reranker 做重排
4. 返回最终的 Top K 个结果

---

### 2.5 Rerank 重排

**知识点讲解：**

Rerank（重排）是在初筛之后，用一个更精确的模型对候选结果重新排序的技术。

**为什么需要 Rerank：**
- 向量检索速度快，但相似度计算比较粗，只是向量点积
- Reranker 模型可以做更精细的语义匹配，比如考虑 query 和 document 的交互信息
- 但 Reranker 计算量大，所以先快速召回一批候选，再用 Reranker 精排，兼顾速度和效果

**典型流程：**
```
Query → 向量检索/全文检索召回 Top 100 → Reranker 重排 → 返回 Top 10
```

**常用的 Rerank 模型：**
- BGE-Reranker（智源）
- bge-reranker-v2-m3
- Cohere Rerank
- DashScope gte-rerank

**在本项目中的应用：**

- 用 DashScope 的 gte-rerank-v2 作为 Reranker
- 混合检索召回后，把候选列表和 query 一起送给 Reranker 重排
- 重排后取 Top K 返回给 RAG 编排器
- 本地开发时用 MockReranker，直接返回原顺序不重排

### 2.6 文本分块（Chunking）

**知识点讲解：**

文本分块就是把长文档切成若干个小段（chunk），每个段单独向量化。这是 RAG 中很关键的一步，分块的好坏直接影响检索效果。

**为什么要分块：**
- 整文档向量太大，语义不聚焦，检索不准
- LLM 上下文窗口有限，不可能把整个文档都塞进去
- 检索时只需要相关的片段，不需要整文档

**常见的分块策略：**
1. **固定长度分块** — 按字符数或 token 数切，简单但可能切断语义
2. **语义分块** — 按段落、句子、标题等语义边界切，效果更好
3. **递归字符分块** — 优先按段落切，太长了再按句子切，还长就按字符切
4. **基于结构的分块** — 针对 PDF、Markdown 等有结构的文档，按标题层级分块

**分块大小怎么选：**
- 太小：上下文信息不够，语义不完整
- 太大：噪声多，检索不准，还占 token
- 经验值：一般 500-1000 个 token 比较常用

**在本项目中的应用：**

- 实现了 `Chunker` 基类和 `PageAwareChunker`（页码感知分块器）
- 优先按段落/换行符切分，超过最大长度再递归切小
- PDF 文档特殊处理：每个 chunk 携带 `page_start` 和 `page_end` 字段，记录页码范围
- 分块后还会做文本清洗：去控制字符、修复连字符、去页眉页脚等

---

### 2.7 SSE 流式输出

**知识点讲解：**

SSE（Server-Sent Events，服务器发送事件）是一种服务器主动向客户端推送数据的技术。它基于 HTTP 协议，服务器返回 `text/event-stream` 格式的响应，客户端可以持续接收服务器推送的消息。

**SSE vs WebSocket：**
- SSE 是单向的，只能服务器推给客户端；WebSocket 是双向的
- SSE 基于 HTTP，自动重连，更简单；WebSocket 需要自己实现重连
- SSE 适合只需要服务端推送的场景（比如流式输出）；WebSocket 适合双向通信（比如聊天、游戏）

**SSE 的格式：**
```
data: 这是一条消息

data: 这是第二条消息
可以换行

event: custom_event
data: 自定义事件类型

: 这是注释，用来做心跳
```

**在本项目中的应用：**

- 问答接口用 FastAPI 的 `StreamingResponse` 返回 SSE 流
- LLM 每生成一个 token，就立刻推给前端
- 前端用 `EventSource` 或 `fetch + ReadableStream` 接收流式数据，逐字渲染
- 流式响应里还会推送引用来源、token 使用量等额外信息

## 3. 文档处理与 PDF 解析

### 3.1 PDF 解析

**知识点讲解：**

PDF（Portable Document Format）是一种复杂的文档格式，它本质上是一个绘图指令集合，描述每个字符、线条、图片在页面上的位置。所以提取 PDF 的文本不像 Word 或 Markdown 那么直接。

**常用的 Python PDF 解析库：**
1. **PyMuPDF（fitz）** — 速度快，功能强，提取文本质量高，还能处理图片、表格
2. **pypdf** — 纯 Python 实现，轻量，跨平台好，但提取质量稍逊
3. **pdfplumber** — 提取表格效果好，适合表格多的 PDF
4. **pdfminer.six** — 功能全，但比较慢
5. **OCR 类（PaddleOCR、Tesseract）** — 处理扫描版 PDF，上面那些库搞不定的

**PDF 解析的难点：**
- 排版复杂：多栏、页眉页脚、页码、脚注
- 连字符：一行末尾的单词被截断带连字符
- 表格：需要识别表格结构，还原成结构化数据
- 扫描件：本质是图片，需要 OCR
- 编码问题：有些 PDF 字体编码不对，提取出来是乱码

**在本项目中的应用：**

- 主后端用 PyMuPDF，备用后端用 pypdf，PyMuPDF 不可用时自动 fallback
- 配置项 `PDF_PARSER_BACKEND` 可以指定用哪个（auto/pymupdf/pypdf）
- 提取的内容包括：
  - 每一页的文本，带页码
  - PDF 元数据：标题、作者、主题、关键词、页数、创建者、生产者
  - 目录（TOC），如果有的话
  - 表格，转成 Markdown 格式（默认关闭，避免噪声）
- 提取后还会做文本清洗

---

### 3.2 文本清洗

**知识点讲解：**

文本清洗是 NLP 和 RAG 中很重要的预处理步骤，把原始文本中的噪声去掉，提高后续处理的质量。

**常见的清洗操作：**
1. **去控制字符** — 去掉 `\x00`、`\x01` 之类的不可见字符
2. **去多余空白** — 多个空格、多个换行合并成一个
3. **修复连字符** — 英文行末 `exam-\nple` 恢复成 `example`
4. **去页眉页脚** — 每页重复出现的头部底部内容
5. **去页码** — 单独的页码数字
6. **统一编码** — 全角转半角、统一换行符等

**为什么文本清洗很重要：**
- 脏文本会影响 Embedding 的质量，导致检索不准
- 不干净的文本喂给 LLM，可能影响回答质量
- 分块时如果有很多噪声，会浪费 token

**在本项目中的应用：**

实现了 `TextCleaner` 类，包含以下清洗步骤：
1. 去除控制字符和不可见字符
2. 修复英文单词的行末连字符
3. 去除每页的页码行（单独成行的数字）
4. 合并多余的空行
5. 统一空格和换行符

---

### 3.3 页码感知分块

**知识点讲解：**

普通的分块是按文本长度切，不关心页码。页码感知分块（Page-Aware Chunking）会在分块的同时记录每个 chunk 对应的页码范围。

**为什么需要页码：**
- 用户查看引用时，可以知道内容在第几页，方便定位
- PDF 文档本身是有页的概念的，保留页码信息更符合用户习惯
- 可以跳转到对应页面预览

**实现思路：**
- 解析 PDF 时，按页提取文本，每页文本带上页码
- 分块时，记录当前 chunk 包含的起始页和结束页
- 如果一个 chunk 跨了多页，就记 `page_start` 和 `page_end`

**在本项目中的应用：**

- 实现了 `PageAwareChunker` 类，继承自 `Chunker`
- 输入是带页码的页面列表 `[{page: 1, text: '...'}, ...]`
- 输出是带页码范围的 chunk 列表 `[{content: '...', page_start: 1, page_end: 2}, ...]`
- chunk 存入数据库时，`DocumentChunk` 表有 `page_start` 和 `page_end` 两个字段
- 前端引用展示时，如果有页码就显示「第 X 页」

## 4. 异步任务处理

### 4.1 为什么需要异步任务

**知识点讲解：**

在 Web 开发中，有些操作耗时比较长，如果放在 HTTP 请求的处理流程里同步执行，会导致：
- HTTP 请求超时（一般服务器超时是 30s~2min）
- 占用 Web 服务器的连接资源，影响并发
- 用户体验差，一直转圈不知道进度

**常见的耗时操作：**
- 大文件处理、上传
- 发送邮件、短信
- 调用第三方慢接口
- 批量数据处理
- PDF 解析 + 向量化

**解决方案：异步任务队列**
- 把耗时任务放到后台队列里
- HTTP 接口立即返回，告诉用户任务已提交
- 后台 worker 慢慢处理任务
- 用户可以通过轮询或其他方式查看进度

**常用的 Python 任务队列：**
- Celery — 最流行，功能全，但比较重
- RQ（Redis Queue）— 轻量简单，基于 Redis
- Dramatiq — 比 Celery 简单，性能也不错
- 自己用 Redis List 实现 — 简单场景够用

**在本项目中的应用：**

- 用 Redis List 实现了一个简单的任务队列
- 有专门的 `IngestionWorker` 进程来消费任务
- 触发异步处理的条件：
  - 文件大小超过 `ASYNC_PROCESS_THRESHOLD_MB`（默认 10MB）
  - 文件是 PDF 格式（不管大小都异步）
- 上传接口立即返回文档 ID，状态是 `pending`
- 后台 worker 处理完后更新文档状态为 `ready` 或 `failed`
- 还记录了 `IngestionJob`，存任务状态、错误信息、重试次数

---

### 4.2 异步任务的可靠性设计

**知识点讲解：**

异步任务不能只是简单地丢到队列里就不管了，还要考虑可靠性：

**关键问题和解决方案：**

1. **任务丢失怎么办？**
   - 任务入队和数据库操作要保证事务一致
   - worker 取任务后不要立刻删，处理完成再确认删除
   - 任务状态持久化到数据库，重启也不会丢

2. **任务执行失败怎么办？**
   - 捕获异常，记录错误信息
   - 设置重试次数，失败了可以重试
   - 超过最大重试次数标记为失败，方便排查

3. **数据一致性怎么保证？**
   - 重要状态变更要及时落库
   - 比如解析完元数据就先存了，不要等全部完成再存
   - 即使后面向量化失败了，至少解析的内容还在

4. **怎么看任务进度？**
   - 任务有状态字段：queued / processing / completed / failed
   - 可以有进度百分比（复杂的话）
   - 失败了有错误信息，方便定位

**在本项目中的应用：**

- `IngestionJob` 表记录每个摄取任务的状态、错误信息、重试次数
- 任务执行前更新状态为 processing，执行成功更新为 completed，失败更新为 failed
- 元数据在向量化之前就先提交到数据库，防止嵌入失败导致解析成果丢失
- Qdrant 和 OpenSearch 的删除操作失败会静默处理，不影响主流程
- 文档状态有 `pending`、`processing`、`ready`、`failed` 四种

## 5. 工程化与性能优化

### 5.1 中间件设计

**知识点讲解：**

中间件（Middleware）是 Web 框架中一个很重要的概念，它可以在请求处理之前和之后插入一些通用逻辑。

**中间件的执行顺序：**
请求进来时从上往下依次经过每个中间件，响应返回时从下往上依次经过每个中间件。

```
请求 → 中间件A → 中间件B → 中间件C → 路由处理函数
响应 ← 中间件A ← 中间件B ← 中间件C ← 路由处理函数
```

**常用的中间件：**
- CORS 中间件 — 处理跨域
- 安全响应头中间件 — 加各种安全相关的 Header
- GZip 压缩中间件 — 压缩响应体，减少传输量
- 限流中间件 — 限制用户请求频率
- 请求 ID 中间件 — 给每个请求加一个唯一 ID，方便追踪问题
- 访问日志中间件 — 记录每个请求的日志
- 认证中间件 — 统一处理登录校验

**在本项目中的应用：**

中间件顺序（从外到内）：
```
CORS → SecurityHeaders → GZip → RateLimit → RequestId → AccessLog
```

每个中间件的作用：
1. **CORS** — 处理跨域请求，配置允许的来源、方法、Header
2. **SecurityHeaders** — 添加 X-Content-Type-Options、X-Frame-Options、Content-Security-Policy 等安全头
3. **GZip** — 压缩响应体，减少网络传输
4. **RateLimit** — API 限流，默认每分钟 60 次，每小时 1000 次
5. **RequestId** — 给每个请求生成唯一 ID，加到日志和响应头里
6. **AccessLog** — 记录访问日志，包含请求方法、路径、状态码、耗时等

---

### 5.2 优雅降级

**知识点讲解：**

优雅降级（Graceful Degradation）是指系统在某些依赖不可用时，不会直接挂掉，而是自动切换到备用方案，保证核心功能还能用。

**为什么需要优雅降级：**
- 生产环境中，某个服务挂掉是常有的事
- 非核心功能挂了不应该影响主流程
- 给用户更好的体验，而不是直接报错

**常见的降级策略：**
- 服务不可用时切换到备用实现
- 缓存兜底，返回旧数据
- 功能开关，临时关闭非核心功能
- 返回默认值或空结果

**在本项目中的应用：**

1. **OpenSearch 不可用 → 降级到 PostgreSQL 全文检索**
   - 启动时检测 OpenSearch 连接
   - 连不上的话，全文检索自动用 PostgreSQL 实现
   - 用户感知不到，只是检索性能和效果稍差一点

2. **Qdrant / OpenSearch 删除失败 → 静默处理**
   - 删除文档时，如果向量库或全文索引删除失败
   - 记录日志，但不抛异常，不影响文档删除的主流程
   - 因为数据主要在数据库里，索引删失败顶多是多几条脏数据，不影响使用

3. **PDF 解析器降级**
   - PyMuPDF 导入失败时，自动 fallback 到 pypdf
   - 虽然解析质量稍差，但至少能用

### 5.3 连接池与单例模式

**知识点讲解：**

**连接池（Connection Pool）：**
每次新建数据库连接、HTTP 连接都是比较耗时的操作。连接池的思想是预先创建一批连接，用的时候从池子里取，用完还回去，不用每次新建销毁。

**连接池的好处：**
- 减少连接创建销毁的开销
- 控制连接数量，防止把数据库打爆
- 提高响应速度

**单例模式（Singleton）：**
单例模式保证一个类只有一个实例，并且提供一个全局访问点。

**什么时候用单例：**
- 配置对象（全局只有一份配置）
- 连接池（全局只有一个连接池）
- 客户端对象（比如 Qdrant 客户端、OpenSearch 客户端，不需要多个）
- 日志对象

**在本项目中的应用：**

1. **数据库连接池**
   - SQLAlchemy 配置：`pool_size=20, max_overflow=10`
   - `pool_recycle=3600` — 连接超过 1 小时自动回收，防止连接失效
   - `pool_pre_ping=True` — 取连接前先 ping 一下，坏了自动重连
   - `pool_use_lifo=True` — 后进先出，常用的连接更可能存活

2. **Qdrant 客户端单例懒加载**
   - 用 `@lru_cache` 或全局变量保证只有一个 Qdrant 客户端实例
   - 第一次使用时才初始化（懒加载），不是启动时就建
   - 避免重复创建连接

3. **OpenSearch 连接池**
   - 用 `RequestsHttpConnection` 连接类
   - `pool_maxsize=10`，连接池大小 10

### 5.4 API 限流

**知识点讲解：**

API 限流（Rate Limiting）是限制用户在单位时间内的请求次数，防止恶意刷接口或误操作把系统打挂。

**常见的限流算法：**

1. **固定窗口计数器** — 按分钟/小时计数，超过就拒绝。简单，但有临界问题（比如 1:59 和 2:00 各发 100 次，其实两秒内发了 200 次）

2. **滑动窗口计数器** — 滑动时间窗口，解决了临界问题，但实现稍复杂

3. **漏桶算法（Leaky Bucket）** — 请求进入桶里，以固定速率流出。流量整形，输出平稳。

4. **令牌桶算法（Token Bucket）** — 桶里有令牌，每秒加一定数量，请求来拿令牌，拿到就放行。可以承受一定的突发流量。

**限流的维度：**
- 按 IP 限流
- 按用户 ID 限流
- 按接口限流

**在本项目中的应用：**

- 用 Redis 实现了滑动窗口限流
- 配置项：
  - `ENABLE_RATE_LIMIT` — 是否开启限流，默认 false
  - `RATE_LIMIT_PER_MINUTE` — 每分钟限制，默认 60 次
  - `RATE_LIMIT_PER_HOUR` — 每小时限制，默认 1000 次
- 按用户 ID 限流（登录用户）或 IP 限流（未登录）
- 超过限制返回 429 Too Many Requests

## 7. 面试常见问题与回答

### 7.1 RAG 相关

**Q1: 简单说一下你理解的 RAG，它解决了什么问题？**

A: RAG 就是检索增强生成，在 LLM 生成回答之前，先从知识库中检索相关的文档片段，把这些片段和问题一起作为上下文喂给 LLM，让 LLM 基于检索到的信息来回答。

它主要解决几个问题：一是 LLM 的知识有截止日期，不知道最新的信息；二是 LLM 容易产生幻觉，RAG 可以让回答基于真实的文档，有据可查；三是企业的私有数据不可能都用来微调，RAG 可以直接对接私有知识库，成本也低很多。

---

**Q2: RAG 的完整流程是怎样的？**

A: 分两个阶段，离线和在线。

离线阶段就是文档入库的过程：用户上传文档，解析文本，做文本清洗，然后分块，每个块生成 Embedding 向量，存到向量数据库里，同时可能也会存一份到全文检索引擎。

在线阶段就是问答的过程：用户提一个问题，先把问题转成向量，去向量库检索相关的 chunk，同时也会做全文检索，然后把两路结果用 RRF 融合，再经过 Reranker 重排，选出最相关的 Top K。接着把这些 chunk 和问题一起组织成 Prompt，发给 LLM 生成回答，最后通过 SSE 流式返回给前端。

---

**Q3: 为什么要用混合检索？只用向量检索不行吗？**

A: 向量检索擅长语义匹配，能找到意思相近但用词不一样的内容，但它对精确的关键词匹配不敏感，比如专有名词、产品型号、人名这些精确的词，可能反而检索不到。

全文检索正好相反，擅长关键词精确匹配，但理解不了语义和同义词。

所以把两者结合起来效果最好，向量检索补语义，全文检索补精确匹配。我们用 RRF 算法融合两路结果的排名，最后再用 Reranker 精排一下。

---

**Q4: RRF 是什么？为什么不用加权平均？**

A: RRF 是倒数排名融合，它给每个结果打分的方式是 1/(k + rank)，k 一般取 60，同一个文档在两路结果里都出现的话分数相加，最后按总分排序。

不用加权平均主要是因为向量检索的相似度分数和全文检索的 BM25 分数不是一个量级的，没法直接加权。向量相似度可能是 0.7 左右，BM25 可能是十几分，权重很难调。RRF 只看排名不看具体分数，就避开了这个问题，实现也简单，效果还稳定。

---

**Q5: 文本分块有哪些策略？你们是怎么做的？**

A: 常见的分块策略有固定长度分块、语义分块、递归字符分块、基于结构的分块等等。

我们用的是递归方式，优先按段落和换行切，超过最大长度再按句子切，还长就按字符切。针对 PDF 我们做了页码感知分块，每个 chunk 会记录起始页和结束页，这样回答里的引用可以精确到第几页，用户可以直接跳转到对应页面查看。

分块大小我们根据经验设的是几百个 token 一块，太大了噪声多检索不准，太小了语义又不完整。分块之前还会做文本清洗，去掉控制字符、修复连字符、去页眉页脚什么的。

---

**Q6: Reranker 有什么用？为什么不直接用向量检索的结果？**

A: Reranker 是重排，就是在初筛之后用一个更精确的模型再排一遍序。

向量检索速度虽然快，但它只是算两个向量的点积，比较粗。Reranker 会把 query 和每个候选文档放在一起做更精细的语义匹配，考虑它们之间的交互信息，排序更准确。但 Reranker 计算量大，不能直接拿它去对全库排序，所以一般是先用向量检索快速召回一批候选，比如 Top 100，再用 Reranker 重排选出 Top 10，这样兼顾速度和效果。

---

**Q7: 怎么评估 RAG 系统的效果？**

A: 评估 RAG 效果一般分几个层面：

一是检索的准确率，就是看检索出来的 chunk 是不是真的相关，可以用命中率、MRR、NDCG 这些指标。

二是回答的质量，就是看 LLM 生成的回答准不准确、完不完整、有没有幻觉。这个可以用人评，也可以用一些自动评估方法，比如用另一个 LLM 来打分。

三是端到端的效果，就是用户满意度，看用户能不能找到想要的答案。

实际做的话，先准备一批测试问题和标准答案，然后分别测检索和回答的指标，上线后再看用户反馈和点击数据。

---

### 7.2 项目架构与设计

**Q1: 你们的系统架构是怎样的？**

A: 我们是前后端分离的架构，前端用 Next.js，后端用 FastAPI，通过 RESTful API 通信，流式问答用 SSE。

后端是模块化设计的，核心的几个组件——Embedding、向量存储、全文检索、LLM、Reranker——都用 Provider 模式封装，可以通过配置切换不同的实现。比如本地开发用 Mock 和内存存储，生产环境切换到真实的服务。

数据存储方面，关系数据存在 PostgreSQL，向量存在 Qdrant，全文索引可以用 PostgreSQL 或者 OpenSearch，Redis 用来做缓存和限流。大文件和 PDF 处理走异步任务队列，后台 worker 处理。

---

**Q2: 为什么要用 Provider 模式？有什么好处？**

A: Provider 模式其实就是策略模式，把不同的实现封装成一个个 Provider，通过配置来选择用哪个。

主要好处是可插拔、易扩展。比如我们的向量存储有 Qdrant 和内存两个实现，本地开发的时候用内存的，不需要起 Qdrant 服务，开发速度快。要上线了改个配置就切到 Qdrant 了。以后想接 Milvus 或者其他向量数据库，只要新增一个 Provider 类就行，业务代码不用动。

另外也方便测试，可以用 Mock 实现替换真实的外部服务，单元测试跑起来又快又稳定。

---

**Q3: 中间件是怎么设计的？有哪些中间件？**

A: 我们的中间件从外到内依次是 CORS、安全响应头、GZip 压缩、限流、请求 ID、访问日志。

CORS 最外层，先处理跨域。然后是安全头，加一些安全相关的 Header。GZip 压缩响应体，减少传输量。再往里是限流，控制请求频率。请求 ID 给每个请求生成唯一标识，方便排查问题。最内层是访问日志，记录请求方法、路径、状态码、耗时这些。

每个中间件各司其职，不互相依赖，顺序也是有讲究的，比如 CORS 要在最外面，不然跨域请求可能被后面的中间件拦掉。

---

**Q4: 你们怎么处理异步任务的？为什么需要异步？**

A: 大文件上传和 PDF 解析加向量化是比较耗时的，如果同步处理的话，HTTP 请求容易超时，用户体验也不好，一直转圈。

所以我们做了异步任务队列，超过 10MB 的文件和所有 PDF 都自动走异步处理。上传接口立刻返回文档 ID 和 pending 状态，后台 worker 慢慢处理。任务状态存在数据库里，有 queued、processing、ready、failed 几种，用户可以看到当前状态，失败了也能看到错误信息。

我们用 Redis List 实现的任务队列，单独的 ingestion worker 进程来消费任务。实现的时候也考虑了可靠性，比如解析完元数据就先落库，不要等全部完成再存，这样即使后面向量化失败了，至少解析的内容还在，可以重试。

---

### 7.3 PDF 与文档处理

**Q1: PDF 解析你们是怎么做的？遇到过什么问题？**

A: 我们用 PyMuPDF 作为主解析器，pypdf 作为备用，PyMuPDF 不可用时自动 fallback 到 pypdf。

遇到的问题还挺多的。比如有些 PDF 有页眉页脚和页码，每页都重复，我们做了文本清洗，去掉了单独的页码行和重复的页眉页脚。还有英文的连字符问题，一行末尾的单词被截断带个横杠，我们做了连字符修复。还有表格的问题，PyMuPDF 有时候会把一些边框装饰识别成表格，产生很多噪声，所以我们默认把表格提取关掉了，需要的时候再开。

另外我们做了页码感知分块，每个 chunk 记录页码范围，这样回答引用的时候可以精确到第几页，用户体验好很多。

---

**Q2: 文本清洗都做了哪些事情？为什么重要？**

A: 文本清洗主要做了这么几件事：一是去掉控制字符和不可见字符，这些东西影响 Embedding 质量；二是修复英文单词的行末连字符，比如 exam-\nple 恢复成 example；三是去掉每页的页码，就是单独成行的数字；四是合并多余的空行和空格。

为什么重要呢，因为脏文本会影响 Embedding 的效果，检索的时候可能就匹配不上了。而且不干净的文本喂给 LLM，也会影响回答的质量，还浪费 token。所以预处理做好了，后面的效果才好。

---

### 7.4 性能优化与稳定性

**Q1: 你们做了哪些性能优化？**

A: 做了不少优化。比如 DashScope 的 Embedding API，我们发现一次传太多文本会返回 400，所以做了分批处理，每批不超过 10 条，虽然慢一点但稳定。

还有 Qdrant 客户端用了单例懒加载，全局只有一个实例，第一次用的时候才初始化，避免重复创建连接。OpenSearch 也配了连接池，pool_maxsize 是 10。

数据库方面，SQLAlchemy 配置了连接池，pool_size 20，max_overflow 10，还开了 pool_pre_ping，连接坏了自动重连。

另外大文件走异步处理，不占用 Web 连接，也避免了 HTTP 超时。GZip 中间件压缩响应，减少网络传输时间。

---

**Q2: 你们的限流是怎么做的？**

A: 我们用 Redis 实现了滑动窗口限流，可以按分钟和小时两个维度限制。配置项可以开关限流，默认是关的，默认每分钟 60 次，每小时 1000 次。

限流维度是按用户 ID，没登录的就按 IP。超过限制返回 429 状态码。用滑动窗口而不是固定窗口，主要是为了避免边界问题，比如 1:59 和 2:00 各发 100 次，其实两秒内发了 200 次，固定窗口就拦不住。

---

**Q3: 优雅降级是什么？你们做了哪些降级？**

A: 优雅降级就是系统某个依赖不可用的时候，不是直接报错挂掉，而是自动切换到备用方案，保证核心功能还能用。

我们做了几处降级。一处是 OpenSearch 连不上的时候，自动降级到 PostgreSQL 的全文检索，虽然效果和性能稍差，但至少检索功能还能用，用户感知不到。还有 Qdrant 和 OpenSearch 的删除操作，如果失败了就记录日志但不抛异常，不影响文档删除的主流程，因为主数据在数据库里，索引有点脏数据问题不大。

另外 PDF 解析也有降级，PyMuPDF 用不了的时候自动切到 pypdf，保证解析功能可用。

---

**Q4: 怎么保证异步任务的数据一致性？**

A: 主要几个点。一是任务状态存在数据库里，IngestionJob 表记录每个任务的状态、错误信息、重试次数，即使 worker 重启了也能知道哪些任务还没处理。

二是状态变更及时落库。比如我们把元数据更新放在向量化之前，解析完文档就把元数据和 chunk 数存到数据库，commit 之后再去做向量化。这样即使向量化失败了，至少解析的内容还在，用户还能看到文档，只是不能检索而已，后面还可以重试。

三是关键操作加错误处理和重试。比如调用外部 API 失败了可以重试几次，不是一次失败就放弃。

---

### 7.5 前端相关

**Q1: 流式输出你们是怎么实现的？**

A: 后端用 FastAPI 的 StreamingResponse，返回 SSE 格式的流。LLM 每生成一个 token，就立刻推给前端，格式是 `data: {content}` 这样的 SSE 标准格式。

前端用 fetch + ReadableStream 来接收，因为 EventSource 只能发 GET 请求，不能带请求体，我们问答接口是 POST 的。收到数据后逐字追加到状态里，实现打字机效果。

除了文本内容，我们还会推送引用来源、token 使用量这些信息，用不同的事件类型区分。

---

**Q2: Markdown 渲染和代码高亮用的什么方案？**

A: 用 react-markdown 渲染 Markdown 内容，代码高亮用的是 react-syntax-highlighter 的 Prism 版本，主题用的 one dark 风格。

代码块我们做了定制，加了语言标签和复制按钮。复制按钮用 navigator.clipboard.writeText 实现，点了之后显示「已复制」的反馈，两秒后恢复。整段回答也有一个一键复制的按钮，放在消息左下角，方便用户复制整段回答。

---

**Q3: 拖拽上传是怎么实现的？**

A: 就是用 HTML5 的拖拽 API。给上传区域绑定 dragover、dragleave、drop 这几个事件。

dragover 的时候要 preventDefault，不然 drop 事件不会触发。拖拽进入的时候给上传区域加个高亮样式，给用户视觉反馈。drop 事件里也要 preventDefault，防止浏览器默认打开文件，然后从 e.dataTransfer.files 拿到文件列表，调用上传接口。

逻辑和点击上传是共用的，只是触发方式不一样。

### 7.6 综合与开放性问题

**Q1: 你在这个项目中最大的收获是什么？**

A: 最大的收获是对 RAG 技术栈有了完整的理解，从文档处理、分块、向量化，到检索、重排、问答编排，整个链路都亲自做了一遍。以前只是知道概念，实际做了才发现里面有很多细节，比如分块的大小怎么选、文本清洗有多重要、混合检索怎么融合效果最好，这些不亲自踩坑是体会不到的。

还有就是工程化方面，以前写代码主要关注功能实现，现在会更多地考虑稳定性、可运维性，比如优雅降级、限流、连接池、异步任务这些，这些东西平时可能不显眼，但生产环境真的很重要。

---

**Q2: 如果让你重新做这个项目，你会有什么不同的做法？**

A: 可能会在几个地方优化。一是评估体系，现在主要靠人测效果，如果重新做的话，我会一开始就搭建一套自动化的评估流程，准备一批测试集，每次优化都跑一遍指标，这样优化效果更直观。

二是可能会考虑用更成熟的框架，比如 LangChain 或者 LlamaIndex，虽然自己实现更灵活，但这些框架提供了很多现成的组件和工具，可以少造一些轮子，开发更快。

三是监控和可观测性，现在只有简单的日志，如果重新做的话，我会加上更完善的监控，比如检索的命中率、回答的质量指标、用户的点击反馈这些，方便持续优化。

---

**Q3: 你觉得 RAG 未来的发展方向有哪些？**

A: 我觉得有几个方向吧。一是多模态 RAG，现在主要是文本，以后图片、表格、音视频这些都可以做 RAG，信息更丰富。

二是 RAG 与 Agent 结合，不只是检索文档，还能根据问题做推理、调用工具、多步检索，解决更复杂的问题。

三是更智能的分块和检索，比如根据文档结构智能分块，或者用 Query 改写、多轮检索这些技术，提升检索准确率。

四是私有化和轻量化，企业对数据安全越来越重视，本地部署的轻量级 RAG 方案需求会越来越大。
