# RAG 主路径体验整改设计

**日期：** 2026-07-07

**目标：** 按真实用户路径整改项目体验问题，同时修复直接影响体验的后端稳定性问题，优先提升“能顺利使用主流程”的质量。

## 背景

当前项目已经具备完整产品雏形，包含登录、知识库、文档管理、搜索、聊天问答、流式输出与引用展示等核心能力。但从实际项目状态看，仍存在几类会直接影响用户体验的问题：

- 前端页面存在部分中文乱码与错误提示不清晰的问题
- 主路径中的加载态、空状态、失败状态不统一，用户容易卡在“不知道系统现在在做什么”
- 文档上传、异步处理、重处理、预览等能力具备，但状态反馈与索引一致性还有风险
- 搜索和问答接口存在契约不完全一致的问题
- RAG 编排与配置读取存在隐式耦合，导致测试与运行行为不稳定

因此，这轮整改不做大范围重构，而是围绕主路径逐段收口。

## 范围

本轮范围限定为一条完整用户主路径：

1. 登录与鉴权
2. 首页与知识库入口
3. 文档上传与处理
4. 搜索
5. 聊天问答

处理原则：

- 优先修复用户直接可见的问题
- 只要某个后端问题会导致报错、卡住、结果不一致或无法验证，就纳入本轮一起修
- 不做与主路径无关的视觉翻新和大规模架构重构

## 方案选型

本轮采用“主路径优先”的整改方式，而不是纯前端先行或纯后端先行。

### 方案 A：主路径优先（采用）

按用户真实操作顺序推进：登录/首页/知识库/文档/搜索/聊天。每一段遇到直接影响体验的后端问题时同步修复。

优点：

- 最贴近用户真实使用过程
- 能同时收敛表层体验问题和根因问题
- 每个阶段都可以独立验收

缺点：

- 需要前后端来回切换
- 中途容易发现额外问题，必须控制范围

### 方案 B：纯前端先行

先统一界面文案、交互状态和错误提示，后续再补后端。

优点：

- 界面反馈会很快改善

缺点：

- 根因不解决时，容易出现“看起来顺了，但功能还是不稳”

### 方案 C：纯后端先行

先修测试、接口契约与 RAG 稳定性，再回头做前端体验。

优点：

- 工程基础更稳

缺点：

- 用户短期感知提升较弱

## 分阶段设计

### 阶段一：入口稳定

**目标：** 用户能顺利登录、进入首页、看到知识库并理解当前状态。

**包含内容：**

- 修复登录页、首页、聊天页、搜索页、知识库页中的中文乱码与明显错误文案
- 统一未登录跳转、鉴权失效处理与错误提示文案
- 优化首页和关键列表页的加载态、空状态、接口失败状态
- 检查前端对当前用户、知识库、会话列表接口的依赖关系，避免首屏因单点失败而整体不可用
- 修复影响阶段一体验的配置耦合问题，保证本地默认行为与测试行为一致

**验收标准：**

- 未登录访问受保护页面会稳定跳转到登录页
- 登录后首页能稳定显示，不出现乱码文案
- 知识库为空、会话为空、接口失败时都有明确反馈
- 后端基础测试可稳定运行，至少不因本地 `.env` 污染导致核心流程误走外部依赖

### 阶段二：资料链路稳定

**目标：** 用户能上传资料，并清楚知道系统是否成功处理。

**包含内容：**

- 优化知识库详情页中的上传、处理中、完成、失败状态反馈
- 检查文档列表刷新、重命名、预览、重处理、内容编辑后的状态回填
- 修复上传后数据库记录、存储、向量索引、全文索引之间的状态不一致问题
- 提高异步任务结果的可见性，让前端能区分 pending、processing、completed、empty、failed
- 处理文档删除或重处理时遗留索引和状态未清理的问题

**验收标准：**

- 上传后文档状态变化可见
- 重处理与编辑后结果能正确反映到预览、检索或问答
- 失败状态可见且不会让页面停留在假“处理中”

### 阶段三：检索与问答体验收口

**目标：** 用户能搜到内容、问出结果，并理解答案来自哪里。

**包含内容：**

- 统一搜索接口前后端契约，明确是否支持知识库过滤以及参数形式
- 优化搜索页结果展示、空结果反馈、错误提示
- 优化聊天页流式回答、引用展示、复制体验与异常提示
- 修复 RAG 回退逻辑与外部 LLM 配置之间的隐式耦合，使默认行为可预测
- 补足核心回归测试，覆盖至少一个“检索到内容并成功回答”的主链路

**验收标准：**

- 搜索行为与 UI 预期一致，不存在前端传参但后端不生效的问题
- 聊天问答在默认配置下可稳定给出回答或可解释的失败信息
- 引用展示与答案结果结构一致

## 架构与实现约束

- 优先沿用现有目录结构与分层方式
- 避免对大型页面进行无关重写，只做与主路径整改相关的局部收口
- 后端优先通过依赖注入、显式参数或更明确的默认行为来解除隐式耦合
- 接口变更时，同步更新前端调用和测试，不保留“双重语义”

## 错误处理设计

- 前端错误提示按三类区分：鉴权失效、用户操作失败、系统异常
- 异步任务状态必须可感知，不允许长期停留在模糊状态
- 当外部依赖不可用时，优先降级到可解释结果，而不是让请求静默失败或无响应

## 测试策略

本轮采用“主路径回归 + 局部单测/接口测试”的组合方式：

- 后端：补齐或修正 RAG 编排、搜索接口、文档状态流转相关测试
- 前端：至少通过构建与关键页面逻辑检查验证主路径未被破坏
- 整体验证：按用户路径手动验证登录、进入知识库、上传、搜索、聊天这条链路

## 非目标

以下内容不在本轮范围内：

- 全量 UI 视觉重设计
- 引入新的基础设施或大规模中间件替换
- 完整 CI/CD 体系重建
- 与主路径无关的性能专项优化

## 成功定义

当以下条件同时满足时，认为本轮整改达标：

- 主路径页面不再出现明显乱码、错误提示缺失或状态混乱
- 用户可从登录一路走到上传、搜索、问答，不会频繁遭遇卡住或不可解释的失败
- 直接影响体验的后端稳定性问题得到收口，核心测试与默认行为更一致
- 搜索与问答的结果结构、引用和前端展示语义保持一致
